## 🔄 ЖИЗНЕННЫЙ ЦИКЛ ЗАПРОСА

### Пример: `GET /api/technologies?search=AI&page=1`

```
1️⃣ КЛИЕНТ
   │
   │ GET /api/technologies?search=AI&page=1
   ↓
2️⃣ ROUTER (router.go:82)
   │
   │ Находит маршрут: api.Get("/technologies", tech.List)
   ↓
3️⃣ HANDLER (technology_handler.go:List)
   │
   │ • Парсит параметры: search="AI", page=1
   │ • Создает domain.TechnologyListParams
   │ • Вызывает service.List(ctx, params)
   ↓
4️⃣ SERVICE (technology_service.go:List)
   │
   │ • Валидирует параметры
   │ • Добавляет defaults (limit=20)
   │ • Вызывает repo.FindAll(ctx, params)
   ↓
5️⃣ REPOSITORY (technology_repo.go:FindAll)
   │
   │ • Строит SQL-запрос с WHERE, LIMIT, OFFSET
   │ • Выполняет db.Query(ctx, sql, params...)
   │ • Парсит результаты в []domain.Technology
   ↓
6️⃣ POSTGRESQL
   │
   │ SELECT id, slug, name, ...
   │ FROM technologies
   │ WHERE name ILIKE '%AI%'
   │ ORDER BY list_index
   │ LIMIT 20 OFFSET 0
   ↓
5️⃣ REPOSITORY
   │
   │ Возвращает: items []Technology, total int, err
   ↑
4️⃣ SERVICE
   │
   │ Возвращает: items, total, nil
   ↑
3️⃣ HANDLER
   │
   │ writeJSON(w, 200, {
   │   "data": items,
   │   "total": total,
   │   "page": 1
   │ })
   ↑
2️⃣ ROUTER
   ↑
1️⃣ КЛИЕНТ
   
   Получает JSON:
   {
     "data": [
       {"id": "...", "name": "Нейросети", ...},
       {"id": "...", "name": "AI ассистенты", ...}
     ],
     "total": 42,
     "page": 1
   }
```

## 📊 АРХИТЕКТУРНЫЕ ПРИНЦИПЫ

### 1. **Separation of Concerns (Разделение ответственности)**

```
┌──────────────────────────────────────────┐
│ HTTPAPI (Presentation Layer)             │
│ • Парсинг HTTP запросов                  │
│ • Валидация входных данных               │
│ • Формирование HTTP ответов              │
└──────────────────┬───────────────────────┘
                   │ domain.Technology
                   ↓
┌──────────────────────────────────────────┐
│ SERVICE (Business Logic Layer)           │
│ • Бизнес-правила                         │
│ • Оркестрация операций                   │
│ • Дополнительная валидация               │
└──────────────────┬───────────────────────┘
                   │ domain.Technology
                   ↓
┌──────────────────────────────────────────┐
│ REPOSITORY (Data Access Layer)           │
│ • SQL запросы                            │
│ • Маппинг БД ↔ Domain модели             │
│ • Работа с транзакциями                  │
└──────────────────┬───────────────────────┘
                   │ SQL
                   ↓
┌──────────────────────────────────────────┐
│ POSTGRESQL (Database)                    │
└──────────────────────────────────────────┘
```

### 2. **Dependency Inversion (Инверсия зависимостей)**

```
❌ ПЛОХО (прямая зависимость):

┌─────────┐      ┌─────────────┐
│ SERVICE │─────→│  PostgreSQL │
└─────────┘      │  Repository │
                 └─────────────┘

Проблема: Service зависит от конкретной реализации.
Нельзя заменить БД или написать тест.


✅ ХОРОШО (через интерфейс):

┌─────────┐      ┌──────────────────┐
│ SERVICE │─────→│    INTERFACE     │
└─────────┘      │ TechnologyRepo   │
                 └────────┬─────────┘
                          ↑
                 ┌────────┴─────────┐
                 │  PostgreSQL      │
                 │  Repository      │
                 └──────────────────┘

Service зависит от интерфейса.
Можно подставить любую реализацию!
```

### 3. **Single Responsibility (Единственная ответственность)**

```
✅ Каждый компонент делает ОДНО:

TechnologyHandler → Обрабатывает HTTP
TechnologyService → Бизнес-логика
TechnologyRepo    → Работа с БД
```
